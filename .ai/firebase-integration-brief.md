# Firebase Integration — запит до розробників BroCalories

Цей документ — короткий бриф для мобільних розробників BroCalories, щоб ми (backend NutriScan) могли інтегруватись з вашим Firebase-проєктом і прив'язувати кожен скан їжі до конкретного користувача.

---

## Що ми хочемо зробити

Зараз `POST /analyze/` від мобільного приходить голим multipart-запитом з фотографією. Ми не знаємо, який саме користувач його зробив — статистика у нашій адмінці тільки сукупна. Хочемо:

1. Бачити **історію сканів кожного юзера** (BroCalories user X → 12 сканів сьогодні, $0.18 cost).
2. Тягнути з вашого Firebase **профіль** (ім'я, email, демографія, статус підписки) для маркетингу і per-user аналітики.
3. (Опціонально) **верифікувати** що `X-User-Id` справді належить тому, хто шле запит — щоб ніхто не міг "підмінитись" під іншого юзера.

Імплементація з боку NutriScan — заплановано як **v1.2.0** (наступний реліз після поточного v1.1.x). Цей файл — список того, що нам треба від вас, щоб ми могли почати.

---

## Що треба з вашої сторони

### 1. Передавати UID разом з фотографією

При `POST /analyze/` додайте у запит HTTP-header:

```
X-User-Id: <firebase_uid>
```

де `firebase_uid` — це звичайний `user.uid` з Firebase Auth (28-символьна строка, наприклад `7Bx9zP3mKqY8wNsA2fLgVcEhRtUi`).

**Опціонально (для безпеки):** замість `X-User-Id` шліть Firebase ID-токен:

```
Authorization: Bearer <id_token>
```

де `id_token = await user.getIdToken()`. Тоді наш backend верифікує токен через Firebase Admin SDK і витягне UID сам. **Перевага:** ніхто не може підмінитись під чужий UID; **мінус:** токен валідний ~1 годину, треба рефрешити (Firebase SDK робить це автоматично).

Скажіть, який варіант обираємо. Я б рекомендував одразу другий — щоб не повертатись пізніше.

### 2. Service Account JSON

Нам потрібен service account з правами читання Firebase Auth + Firestore. Як згенерувати:

1. Firebase Console → **Project Settings** (шестерінка) → **Service accounts** tab
2. Натиснути **"Generate new private key"**
3. Завантажиться файл `<project>-firebase-adminsdk-XXXXX-YYYY.json` (~2-3 KB)
4. Передайте його нам **захищеним каналом** (Signal, шифрований email, Bitwarden Send) — це **приватний** ключ, не committити в git ніде

Ми покладемо файл у `data/firebase-service-account.json` на сервері (тільки на проді, `.gitignored`).

### 3. Опишіть структуру `users/{uid}` document у Firestore

Підкажіть, які поля ви зараз зберігаєте у Firestore-документі `users/{uid}` (або інша колекція з профілем). Конкретно нас цікавить:

**Базове (для розпізнавання):**
- `email` — мабуть є у Firebase Auth уже
- `displayName` / `name` / `first_name + last_name`
- `photoURL` / `avatar_url`
- `createdAt` / `registered_at`

**Демографія (для маркетингу — якщо ви це збираєте):**
- `gender` — male / female / other / not-specified
- `age` або `date_of_birth`
- `country` / `region` / `city`
- `language` / `locale`
- `weight_kg`, `height_cm` — якщо це фітнес-додаток і ви це питаєте при онбордингу
- `daily_calorie_goal`, `goal` (lose / maintain / gain) — теж для онбордингу

**Підписка (для v1.3 tier-based model selection):**
- `subscription_tier` — наприклад `free` / `premium` / `pro`
- `subscription_expires_at`
- `is_trial`

**Інше:**
- `referral_source` — як знайшли додаток (Google Play, ads, friend)
- `last_active_at`
- Custom Claims (Firebase Auth) — якщо щось туди записуєте (роль, tier тощо)

Просто покажіть який зразок документу — ми оберемо що нам корисно і будемо це тягнути.

### 4. Тестовий UID

Дайте один-два тестових UID (юзери з вашого dev-середовища або просто реальні юзери з демо-профілями), щоб ми могли локально перевірити що інтеграція працює. Без живого UID ми не можемо протестувати.

### 5. Firebase Project ID

Просто скажіть назву (наприклад `brocalories-prod`). Це не секретна інформація.

## Що відбувається на нашій стороні (для прозорості)

Після того як ви все надасте, наш backend почне:

1. На кожен `POST /analyze/` витягувати `mobile_user_id` з header або з токена.
2. Зберігати в нашій SQLite-БД у колонку `requests.mobile_user_id`.
3. Періодично (з кешем TTL 24 години, асинхронно — не блокує відповідь) тягнути профіль з Firebase і кешувати локально в таблицю `mobile_users`.
4. У нашій адмінці буде сторінка `/users` зі списком усіх ваших юзерів та drill-down `/users/<uid>` з повною статистикою (скільки фоток, який cost, які страви).

**Жодних даних ми мобільному додатку не повертаємо назад** — все це для внутрішньої аналітики/моніторингу та для per-user обмежень у майбутньому.
